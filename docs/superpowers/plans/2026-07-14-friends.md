# Friends Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a unified, privacy-aware Friends hub for Game Finder users, Steam contacts, and manual PSN contacts, then deploy it safely on Railway.

**Architecture:** Persist the site social graph and profile settings in PostgreSQL; merge it at read time with bounded, cached Steam data and manual PSN contacts. FastAPI applies privacy before serializing data. Next.js consumes typed API responses in a new Friends hub, profile route, and account settings panel. A one-replica Railway worker refreshes Steam snapshots, while API migrations run as a pre-deploy command.

**Tech Stack:** FastAPI, SQLAlchemy 2, Alembic, PostgreSQL, Redis, httpx, Next.js 16, React 19, TypeScript, Vitest/React Testing Library, Railway Docker services.

## Global Constraints

- `public_nickname` is normalized case-insensitively, unique, and email is never emitted in social responses.
- Activity, platforms, and game data default to `everyone`; each supports `friends` and `nobody`; server-side filtering is mandatory.
- Steam matching may enrich a card but never creates a Game Finder friendship.
- PSN has no API calls: accept a validated Online ID and construct only a fixed canonical outbound URL.
- Reusable invites store only a cryptographic token digest; accepting requires authentication and confirmation.
- Railway migrations run only via backend pre-deploy `python -m alembic upgrade head`; social refresh is a separate single-replica worker.
