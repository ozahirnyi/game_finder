# Исправление `Data unavailable` и профиля — план для нового чата

## Что произошло

Текущий дизайн подключён, но страницы профиля и главной используют лишь часть API и для остальных полей подставляют строку `Data unavailable`. Это не означает, что весь backend не работает: каталог, цены, библиотека и Steam уже имеют маршруты, но их результаты не собираются в модель данных интерфейса. Отдельно был исправлен вход: на Railway опубликованы реальные `/login` и `/register`, а заглушечный профиль в меню убран.

## Ключевые изменения

- Добавить защищённые сводные API:
  - `GET /dashboard` — данные главной страницы: пользователь, библиотека и её статистика, персональные рекомендации, сделки/изменения цен, Steam-статус и социальная активность.
  - `GET /profile/summary` — данные профиля: аккаунт, подключённые сервисы, статистика библиотеки, избранное, wishlist и недавно сыгранные игры.
  - Каждый блок возвращает явный статус `ready`, `empty`, `not_connected` или `error`; frontend не выводит технический текст `Data unavailable`.

- На backend переиспользовать существующие сервисы каталога, цен, `/games`, Steam, PSN, Telegram и OAuth. Новые ответы собрать в отдельные схемы, без дублирования бизнес-логики.
- Для полей профиля, которых пока нет в БД (bio, платформы, любимые жанры), добавить сохраняемый профиль пользователя и API обновления. Не выдумывать данные: заполненные поля отображаются, незаполненные предлагают настроить профиль.
- Переподключить страницы главной, профиля, библиотеки и Steam к реальным данным:
  - убрать все статические `Data unavailable`, фальшивую активность и демо-статистику;
  - отображать реальные игры, часы, статусы сохранённых игр, Steam-библиотеку, цены и рекомендации;
  - использовать полезные пустые состояния с кнопками «Подключить Steam», «Импортировать PSN», «Добавить игру», «Настроить профиль»;
  - сохранить текущий дизайн Lovable, изменяя только источники и состояния данных.

- Привести авторизацию к единому поведению: отсутствие токена ведёт на `/login`; после входа профиль и защищённые запросы обновляются без перезагрузки; истёкший токен очищается корректно.
- Удалить или изолировать устаревшие Next.js-страницы, если они не участвуют в TanStack/Vite-приложении, чтобы они не ломали общий тестовый запуск.

## Проверка

- Backend: тесты контрактов для `/dashboard`, `/profile/summary`, профиля и всех статусов блока; проверки доступа без токена и с токеном.
- Frontend: тесты успешных, пустых, неподключённых и ошибочных состояний без строки `Data unavailable`.
- Ручная проверка на Railway:
  1. регистрация и вход;
  2. профиль без подключённых сервисов;
  3. добавление игры в библиотеку;
  4. подключение и синхронизация Steam;
  5. отображение каталога, цен и рекомендаций;
  6. обновление страницы и восстановление сессии.
- Перед деплоем: production build frontend, backend test suite, затем публикация обеих служб Railway и проверка публичного URL.

## Принятые решения

- Использовать сводные API для экранов (`/dashboard`, `/profile/summary`), а не собирать десятки запросов на клиенте.
- Если реальных данных нет, показывать полезное пустое состояние, а не заглушку.
- Данные из внешних сервисов не подменять демо-контентом: при недоступности отображать понятное действие или корректную ошибку.
